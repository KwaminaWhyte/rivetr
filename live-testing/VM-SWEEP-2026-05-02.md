## Bugs

### B1 ‚Äî Signup password placeholder/validator mismatch (low/UX)
- Setup form placeholder: "Min 8 characters". Server rejects with "Password must be at least 12 characters".
- Repro: `POST /api/auth/setup` with an 8-character password ‚Üí 400. Frontend doesn't pre-validate length, so the user fills, submits, gets a single-line server error.
- Suspected file: `frontend/app/routes/setup.tsx` (placeholder + missing zod min) and `src/api/auth/*` (12-char rule).
- Fix: change placeholder to "Min 12 characters" and add client-side `min(12)` zod validation.

### B2 ‚Äî `dockerfile` field defaults to `./Dockerfile` even when build_type=nixpacks (cosmetic)
- API created an app with `build_type: "nixpacks"` and `dockerfile: "./Dockerfile"` despite Nixpacks not using a Dockerfile.
- General tab still renders `Dockerfile: ./Dockerfile`. Confusing ‚Äî looks like the app will use a Dockerfile that doesn't exist.
- Repro: Create app via UI ‚Üí choose Nixpacks ‚Üí check `GET /api/apps/:id` and the General tab.
- Suspected file: `src/api/apps/crud.rs` (default-fill on create) + `frontend/app/routes/apps/$id/_index.tsx` (renders dockerfile unconditionally).
- Fix: don't default `dockerfile` when build_type ‚âÝ "dockerfile"; hide field on General tab unless dockerfile build.

### B3 ‚Äî `/api/apps/:id/insights` returns 503 on every page load (medium)
- `cat console.log` shows `503 Service Unavailable` on every app/route navigation.
- Body: `{"error":"AI provider not configured. Add [ai] section to rivetr.toml."}` returned with status **503**.
- Wrong status code ‚Äî 503 means transient unavailability. Should be `200 {ai_configured: false, insights: []}` or `404 {feature: "ai_insights", configured: false}`. As is, every "AI insights not configured" instance log shows up as ERROR in `journalctl -u rivetr` (`tower_http: response failed classification=Status code: 503`), poisoning real-error grep.
- Frontend already renders "AI not configured" correctly ‚Äî but it issues the network call regardless.
- Fix: gate the request on `system/version.ai_enabled` flag, or downgrade to 200/404.
- Suspected file: `src/api/apps/insights.rs` and `frontend/app/components/ai-insights.tsx`.

### B4 ‚Äî Container monitor health check broken every 30 s (HIGH)
- `journalctl` produces continuous warnings:
  `WARN rivetr::engine::container_monitor::health: Failed to fetch running services error=no column found for name: public_access`
- Service health monitoring is silently disabled ‚Äî failed/crashed Compose services won't be detected.
- Root cause: `src/engine/container_monitor/health.rs` ~line 524 SELECT list omits `public_access`, `external_port`, `expose_container_port` columns added by migration 105, but the `Service` struct used in `query_as` requires them. Rivetr v0.10.19 crashes deserialization on every poll.
- Fix snippet (for `check_services`):
  ```sql
  SELECT id, name, project_id, team_id, compose_content, domain, port, status,
         error_message, created_at, updated_at,
         COALESCE(isolated_network, 1) AS isolated_network,
         COALESCE(raw_compose_mode, 0) AS raw_compose_mode,
         COALESCE(public_access, 0) AS public_access,
         COALESCE(external_port, 0) AS external_port,
         COALESCE(expose_container_port, 0) AS expose_container_port
  FROM services WHERE status = 'running'
  ```
- This is a real regression ‚Äî needs immediate patch.

### B5 ‚Äî Misleading "Could not provision MySQL/MariaDB user after 30 s" warning (medium)
- After creating any MySQL or MariaDB DB, a WARN log appears:
  `Could not provision MySQL/MariaDB user after 30 s ‚Äî the app may fail to connect until the database container is restarted`
- Despite the warning, the user actually IS provisioned ‚Äî by the official MariaDB/MySQL entrypoint via `MARIADB_USER` / `MYSQL_USER` env vars ‚Äî and connections work. Verified `SELECT user, host FROM mysql.user;` shows `user_43355` exists.
- The log entry implies broken behaviour; users will think their DB is broken.
- Suspected file: `src/api/databases.rs` (post-start provisioning routine) ‚Äî the "create user" attempt races with the entrypoint and then logs the failure even though the entrypoint succeeds.
- Fix: silence the warning if the user already exists, or skip the racing provisioning step entirely.

### B6 ‚Äî MySQL 8 connection requires SSL but `internal_connection_string` doesn't say so (medium)
- DB API returns `mysql://user:pass@host:3306/db` with no SSL hint.
- Connecting via the official `mysql:8` client fails: `ERROR 2026 (HY000): TLS/SSL error: self-signed certificate in certificate chain`.
- Workaround: append `?useSSL=false` / `--ssl=0` / `--skip-ssl`. MariaDB connects fine without that.
- Fix: either disable SSL on the MySQL container init (matches MariaDB), or include `?ssl_mode=DISABLED` in the connection string.

### B7 ‚Äî Audit log misses many event types (medium)
- Captured: `auth.login`, `auth.setup`, `project.create`, `app.create`, `database.create`, `deployment.trigger`.
- Not captured: `token.create`, `token.delete`, `app.start/stop/restart`, `app.delete`, `app.update`, `service.create/start/stop`, `database.delete`, `deployment.cancel`, `deployment.rollback`, `env_var.create/update/delete`, `domain.add/remove`.
- Repro: do any of the above, then `GET /api/audit?per_page=50` ‚Äî entry not present.
- Suspected file: `src/api/audit.rs` + each missing event handler in `src/api/{apps,databases,services,api_tokens,deployments}/*.rs`.

### B8 ‚Äî Audit `ip_address` is always `null` (low/security)
- All entries return `"ip_address": null`. Useful for security forensics, currently dead.
- Suspected file: `src/api/audit.rs` (insertion never reads `X-Forwarded-For` or `ConnectInfo`).

### B9 ‚Äî Audit `?limit=N` query param is ignored (low)
- Tested: `GET /api/audit?limit=2&page=2` ‚Üí response `per_page: 50, items_count: 0`.
- Server only accepts `per_page=N`. Inconsistent with most REST APIs; CLI/SDK users will assume `limit` works.
- Fix: accept either alias.

### B10 ‚Äî DELETE / POST endpoints reject empty/no Content-Type (medium)
- `DELETE /api/apps/:id` returns 415 without `Content-Type: application/json`, then 400 "EOF while parsing" with header but no body.
  Final correct call: `DELETE ‚Ä¶ -H 'Content-Type: application/json' -d '{"password":"‚Ä¶"}'`.
- `POST /api/deployments/:id/rollback` returns 415 without `Content-Type` even though it accepts an empty body.
- `POST /api/apps/:id/deployments/:id/cancel` similar.
- Other Rivetr POSTs (start/stop/restart/redeploy) accept no body / no header just fine.
- Inconsistency makes raw curl/SDK use error-prone.
- Fix: either accept empty body universally, or document required headers in `/api/sdk` TS file and the API reference.

### B11 ‚Äî Cancel deployment returns 404 instead of 409 in non-cancellable states (low)
- `POST /api/apps/:id/deployments/:id/cancel` for an already-running deployment returns:
  `404 {"message":"Deployment not found or is not in a cancellable state"}`.
- 404 is wrong (deployment exists). Should be 409 Conflict, 422 Unprocessable, or 200 with `{cancelled:false, reason:"..."}`.
- Suspected file: `src/api/deployments/control.rs` (`cancel_deployment`).

### B12 ‚Äî Rollback commit_message references the wrong deployment (cosmetic)
- After `POST /api/deployments/<old>/rollback`, the new deployment row has:
  `commit_message: "Rollback to deployment <previous-running-id>"`
- It actually rolls back TO `<old>` (the path param). The message should be `"Rollback to deployment <old>"`.
- Suspected file: `src/api/deployments/rollback.rs` (variable swap when constructing message).

### B13 ‚Äî Old replaced deployment marked `failed` after rollback (medium)
- Deployment 408c‚Ä¶ was `running`; after rolling back to a prior deployment, 408c‚Ä¶ became `failed`.
- The very first deployment in this app's history was correctly marked `replaced` after the second deploy. The post-rollback transition should also be `replaced` or `superseded` ‚Äî `failed` implies a build error that didn't happen.
- Suspected file: `src/engine/pipeline/finalize.rs` or `src/api/deployments/rollback.rs` (status transition during rollback).

### B14 ‚Äî Restart leaves a deploy-id-suffixed container name (cosmetic, regression-prone)
- Before restart: `rivetr-hello-node`. After zero-downtime restart: `rivetr-hello-node-restart-173a2022`.
- The suffix sticks until the next deployment. Two consequences:
  1. Network tab UI shows `Container Name: rivetr-hello-node-274d7136` with a code snippet `curl http://rivetr-hello-node-274d7136:5000` ‚Äî this hostname is wrong after every redeploy.
  2. Users running `docker ps` get inconsistent naming.
- Fix: after the new container becomes healthy and the old one is torn down, rename the new container to the canonical `rivetr-<app-name>`. Or set a stable network alias on every container (see B15).

### B15 ‚Äî Service-to-service hostname examples are unstable (UX/medium)
- Network tab connection example uses `rivetr-hello-node-274d7136`, which changes every deploy.
- Suspected fix: ensure each app container has a stable Docker network alias matching the app name (e.g. `rivetr-<app-name>`), then surface that in the example. The model `App` should expose an `internal_hostname` derived field.

### B16 ‚Äî Network tab labels Docker network "rivetr-network" but it's "rivetr" (cosmetic)
- `docker network ls` shows `rivetr` (and per-app `rivetr-app-<id>`). UI says `rivetr-network`.
- Suspected file: `frontend/app/routes/apps/$id/network.tsx` (hardcoded string).

### B17 ‚Äî Templates list is 500 KB uncompressed, served on every Templates page render (PERF/medium)
- `GET /api/templates` returns 322 templates with full `compose_template` + `env_schema` strings inline ‚Üí ~500 KB JSON.
- Server does NOT honor `Accept-Encoding: gzip` (`content-encoding` header absent; response identical size). Fast on LAN; painful over residential uplink to remote dashboard.
- Headers also include `cache-control: no-store, max-age=0` ‚Äî defeats browser caching for an effectively-immutable resource.
- Fixes:
  - Enable a CompressionLayer (Tower middleware) for API responses, or at least serve gzip for application/json above a threshold.
  - Split `compose_template` and `env_schema` into a separate `GET /api/templates/:id` and only ship summaries (id, name, description, icon, category) on the list endpoint.
  - Use `cache-control: public, max-age=300, stale-while-revalidate=900` for the templates endpoint, or add an `ETag`.

### B18 ‚Äî Templates page renders ~1300 cards at once (PERF/medium)
- `document.querySelectorAll('[class*=card]').length === 1288` on first paint (322 templates √ó 4 wrapper cards each).
- No virtualization or infinite scroll. On low-end devices the list scrolls jankily.
- Suspected file: `frontend/app/routes/templates/_index.tsx`.
- Fix: virtualize via `@tanstack/react-virtual` or paginate by category tab.

### B19 ‚Äî Recharts container has invalid initial dimensions on /monitoring (low)
- Console: `The width(-1) and height(-1) of chart should be greater than 0 ‚Ä¶`
- Caused by parent `<ResponsiveContainer>` lacking explicit `min-width` / `min-height` or `aspect`.
- Suspected file: `frontend/app/components/charts/area-chart.tsx`.

### B20 ‚Äî Disk usage reported inconsistently (low)
- Dashboard "Disk Usage" card: `13.14 GB / 61.66 GB (26.4%)` ‚Äî measures `/`.
- Monitoring page: `18.05 GB / 61.66 GB (34.4%)` ‚Äî measures `/var/lib/rivetr` after a couple of DB pulls.
- Both are correct measurements, but two different mountpoints/paths with different numbers in the same UI. Pick one (likely `/var/lib/rivetr` since that's the data dir users actually care about).
- Suspected files: `src/api/system/stats.rs` (Dashboard `/api/system/disk` returns `/`) vs `src/api/system/health.rs` (Monitoring uses `/var/lib/rivetr`).

### B21 ‚Äî `Setup` redirects to `/login` instead of auto-logging-in (UX low)
- After `POST /api/auth/setup` succeeds, frontend pushes user to `/login` and forces them to type the just-typed credentials again.
- Fix: `setup` response could include a session cookie or token; frontend then routes to `/`.

### B22 ‚Äî Deployment "Finished" timestamp is set while status is "Running" (UX low)
- Deployment detail page shows `Finished 2:24:49 PM` even though the app is still running and proxy is live. "Finished" connotes a terminal state and may suggest a stopped service.
- Better label: `Deployed at` / `Started at` for running deployments; `Finished at` only for `replaced`, `failed`, `cancelled`.

### B23 ‚Äî Many UI buttons render twice in DOM (mobile + desktop variants both visible) (UX low)
- On project page: `Create New App`, `Create Database`, `Deploy Template` each appear twice in `querySelectorAll('button, a')`.
- Both are visually rendered (likely a `lg:hidden` / `lg:block` split that didn't actually hide one breakpoint).
- Suspected file: `frontend/app/routes/projects/$id/_index.tsx`.
- Side effect: pressing Tab cycles through duplicates; screen reader announces every button twice.

### B24 ‚Äî Form inputs lack `autocomplete` attributes (a11y low)
- Browser DevTools warning on `/setup` and `/login`: `Input elements should have autocomplete attributes (suggested: "new-password" / "current-password")`.
- Password manager prompts get confused; iOS strong-password generator doesn't trigger.
- Suspected files: `frontend/app/routes/setup.tsx`, `frontend/app/routes/login.tsx`.

### B25 ‚Äî DB-to-app env injection is manual (FEATURE GAP / medium)
- After creating PG/Redis/MySQL/Maria in the same project as an app, the app's container env did NOT receive `DATABASE_URL` / `REDIS_URL`.
- No "Link database" UI exists. Users have to manually copy `internal_connection_string` into Env Vars.
- Compared with Coolify, Railway, Render, etc. which auto-inject. Memory note (`CLAUDE.md`) claimed "Env var injection confirmed", but this only refers to the DB container's own env (POSTGRES_USER etc.), not propagation to apps.
- Improvement: add a "Linked databases" section on the app Env Vars tab; on link, inject `<PREFIX>_URL`, `<PREFIX>_HOST`, `<PREFIX>_PORT`, `<PREFIX>_USER`, `<PREFIX>_PASSWORD`, `<PREFIX>_DB` (default prefix derived from db type).

### B26 ‚Äî Database backup download has wrong Content-Type (cosmetic)
- `GET /api/databases/:id/backups/:bid/download` returns `application/octet-stream` for an SQL plaintext backup.
- Suggest `application/sql` or `text/plain; charset=utf-8` so browsers display rather than save-as a binary.
- Suspected file: `src/api/database_backups.rs:223`.

### B27 ‚Äî Compose service does not auto-generate a domain (medium / feature gap)
- Created Uptime-Kuma via template; `domain` field is `null`. Service is reachable only via host port `:3001`.
- Roadmap note in `TEST-STATUS.md` row 38 mentions auto-domain (commit `9a26dbc`) ‚Äî not happening on a fresh v0.10.19 service create. Either the auto-domain logic depends on a system `instance_domain` setting that's still empty in `white-label`, or it's broken.
- Fix: when system `instance_domain` is blank, fallback to `<service>.local` to give users a working hostname; otherwise generate `<service>.<instance-domain>` automatically.

## Performance issues

- **P1** Templates list endpoint: 500 KB / no gzip / `cache-control: no-store`. Local time 8-15 ms; on 10 Mbps links that's ~400 ms. (See B17.)
- **P2** Templates page first paint creates 1288 DOM nodes; on this VM (aarch64 emulation) initial render is visibly janky for ~300 ms. (See B18.)
- **P3** `/api/system/stats/history?hours=24` returns OK but with no apps it still serializes a full 24-hour grid. With 0 services, payload could be `{}` or 304-cacheable.
- **P4** `cargo zigbuild` not relevant here, but the binary is `aarch64-unknown-linux-gnu` (Parallels ARM) ‚Äî confirms cross-compile target works on Apple Silicon hosts. RAM steady-state ~10 MiB matches advertised "30 MiB idle vs Coolify 800 MiB". With 4 DBs + 1 app + 1 Compose service running concurrently, memory_used_bytes hit 1.1 GiB (mostly Postgres + MariaDB containers); Rivetr itself stayed sub-30 MiB.
- **P5** Database creation is sequential at the pull step ‚Äî three concurrent POSTs to `/api/databases` for distinct images (Redis, MySQL, MariaDB) finished pulling 0 ‚Üí 80 s with the same image-pull semaphore. Probably intentional (avoid disk thrash), but on parallel creates only one image pulls at a time. Could surface this in the UI ("queued: pulling MySQL").
- **P6** Static asset bundle: 90+ JS chunks loaded on first dashboard hit (per network requests log). Likely fine due to embedded HTTP/2 in Axum, but worth verifying. React-Router lazy-loads each route, then loads sub-chunks per component.

## Improvement ideas

1. **Database-to-app linking UI** (see B25) ‚Äî biggest UX gap relative to competitors. One-click "Inject env vars from <db>" ‚Üí adds prefixed env vars + warns redeploy required.
2. **AI insights flag** ‚Äî add `ai_enabled: bool` to `/api/auth/me` or `/api/system/version`; gate the `insights` request and the panel render on it. Eliminates B3 503 noise.
3. **Stable container hostnames** ‚Äî set `--network-alias rivetr-<app-name>` on every app container and surface it in the Network tab. Fixes B14, B15, B16.
4. **Auto-domain for services** ‚Äî fallback to `<name>.<instance-domain | "local">` on create; expose under "Network" tab.
5. **Health-check query patches** ‚Äî pin all `query_as` SELECTs to use `*` or a generated column list from sqlx macros so adding a column never desyncs (preferred long-term).
6. **Standardize HTTP semantics** ‚Äî accept empty body on POST/DELETE endpoints that don't mutate data (start/stop/restart/cancel/rollback/delete-app-when-no-password-required), 409 instead of 404 for non-cancellable states, 200/empty instead of 503 for "AI off".
7. **Audit log coverage matrix** ‚Äî write a list of events that should be audited and unit-test that each handler emits one. Probably a 30-line proc-macro that wraps audited handlers.
8. **Templates endpoint slimming** ‚Äî split list (summary) vs detail. Strip `compose_template` and `env_schema` from list, plus `Cache-Control: public, max-age=300`. Will reduce the endpoint to <30 KB.
9. **Compression middleware** ‚Äî add `tower-http` `CompressionLayer` for JSON responses ‚â•4 KB. Free win.
10. **Audit pagination consistency** ‚Äî accept `limit` alias for `per_page`, document in OpenAPI.
11. **Setup ‚Üí auto-login** ‚Äî return session cookie / token on setup so user lands on dashboard without retyping.
12. **Deployment status taxonomy** ‚Äî distinguish `running` vs `succeeded`, `replaced` vs `failed` after rollback. Currently overloaded.
13. **Strip ANSI in log render** ‚Äî use a tiny renderer (`ansi-to-html` or just a regex) in `frontend/app/components/deployment-logs.tsx`.
14. **Hide irrelevant fields per build_type** ‚Äî only show "Dockerfile path" when build_type=dockerfile, only "Static publish dir" for static, etc. Less form clutter.
15. **First-run telemetry** ‚Äî empty Dashboard could surface a "Quick Start" wizard (Project ‚Üí App ‚Üí Deploy your first repo) modal driven by `apps.length === 0 && projects.length === 0`.
